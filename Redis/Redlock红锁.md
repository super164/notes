## Redlock（红锁）

### 一、背景：为什么需要红锁？

在分布式系统中，常用 Redis 实现分布式锁。最简单的做法是单节点执行：

```
SET lock_key unique_value NX PX 30000
```

这个方案存在两个致命问题：

1. **单点故障**：如果唯一的 Redis 节点宕机，整个加锁服务不可用，所有客户端都无法获取锁。
2. **主从异步复制导致锁丢失**：即使部署了主从 + 哨兵，由于复制是异步的，客户端 A 在主节点加锁成功后，主节点在同步锁数据到从节点之前宕机，从节点被提升为新主节点。此时新主节点没有锁记录，客户端 B 可以再次加锁成功，导致两个客户端同时持有锁，造成并发安全问题。

为了解决主从架构下的锁丢失问题，Redis 作者 Salvatore Sanfilippo（Antirez）提出了 **Redlock** 算法。

------

### 二、红锁的核心思想

红锁**不使用主从复制**，而是部署 **N 个独立的 Redis Master 节点**（通常 N=5）。客户端加锁时，需要向**所有节点**发送加锁请求，只有当**超过半数节点（N/2+1）** 返回成功，且总耗时小于锁有效期时，才认为加锁成功。

> **多数派原则**：任意两个成功的加锁请求，它们的节点集合必然有交集，交集节点会拒绝第二个请求，从而保证互斥。

------

### 三、红锁的加锁流程（详细步骤）

假设有 5 个独立 Redis 节点（A、B、C、D、E），锁超时时间 `TTL = 10 秒`。

1. **获取当前时间**（毫秒级）`start_ms`。

2. **依次向所有节点请求锁**：
   对每个节点执行：

   ```
   SET my_lock random_value NX PX 10000
   ```

   - `my_lock`：锁的名称（标识共享资源）。
   - `random_value`：全局唯一随机值（如 UUID），用于安全释放锁。
   - `NX`：只有键不存在时才设置（保证互斥）。
   - `PX 10000`：锁自动过期时间 10 秒。
   - 每个请求设置**极短的网络超时**（例如 5~50 毫秒），避免某个故障节点长时间阻塞。

3. **计算加锁总耗时**：`elapsed = now_ms - start_ms`。

4. **判断加锁是否成功**：
   需同时满足两个条件：

   - 成功节点数 ≥ **3**（5 个节点中的多数派）。
   - `elapsed < TTL`（加锁过程没有超过锁的有效期）。
   - 如果成功，则**锁的真正有效时间** = `TTL - elapsed`。客户端可在该时间内安全操作共享资源。
   - 如果失败（成功节点数不足或总耗时超时），则立即向**所有节点**发送释放锁请求（执行 Lua 脚本，验证 `random_value` 后删除）。

5. **释放锁**：
   无论加锁成功与否，释放时都需要向**所有 5 个节点**发送释放脚本：

   ```lua
   if redis.call("get", KEYS[1]) == ARGV[1] then
       return redis.call("del", KEYS[1])
   else
       return 0
   end
   ```

### 四、关键设计解析

| 设计点                | 作用                                                 |
| :-------------------- | :--------------------------------------------------- |
| **多个独立节点**      | 避免单点故障，且没有主从复制，消除异步复制丢锁问题。 |
| **多数派同意**        | 保证任意两个成功加锁的请求集合有重叠节点，实现互斥。 |
| **随机值 + Lua 释放** | 防止客户端误删别人持有的锁（例如锁过期后自己再删）。 |
| **极短网络超时**      | 快速跳过故障或慢节点，防止单个节点拖死整体加锁过程。 |
| **锁有效期剩余计算**  | 确保加锁消耗的时间不会侵占业务操作时间。             |

------

### 五、优点

- **解决了主从异步复制导致的锁丢失问题**：不再依赖复制，每个节点独立记录锁状态。
- **高可用**：只要多数节点存活，锁服务仍可用。
- **自动过期**：每个锁都有 TTL，避免客户端崩溃后锁永久不释放。

------

### 六、缺点与争议

尽管红锁提升了安全性，但它并非完美，主要争议来自分布式系统专家 Martin Kleppmann：

1. **时钟跳变问题**
   红锁依赖客户端本地时间计算 `elapsed` 和剩余有效期。如果某个 Redis 节点的系统时钟发生跳变（例如 NTP 修正），可能导致锁提前过期或延后过期，破坏互斥。
2. **GC 暂停 / 网络延迟**
   客户端加锁后可能发生长时间的 GC 暂停（Java）或线程调度延迟，导致实际执行业务的时间超过了锁的有效期，但客户端自己并不知道，此时另一个客户端可能获取锁，造成并发写入。
3. **节点重启导致锁丢失**
   如果获得锁的多数派节点同时宕机并立即重启（内存数据丢失），重启后这些节点不再有锁记录。另一个客户端可以重新获得锁，导致两个客户端同时持有锁。
   **缓解措施**：延迟重启（节点重启后等待一个锁超时时间再对外服务），但无法 100% 消除风险。
4. **复杂度高**：需要维护 5 个独立 Redis 节点，加锁/释放需多次网络 IO，性能低于单节点锁。
5. **并非线性一致**：红锁不能提供像 etcd/ZooKeeper 那样的强一致性保证（如 fencing token），在极端异常下仍可能出现互斥失效。

------

### 七、适用场景建议

| 场景                                           | 推荐方案                                                     |
| :--------------------------------------------- | :----------------------------------------------------------- |
| 大多数业务（防止重复提交、定时任务去重、限流） | 单 Redis 或主从 + 普通锁（如 `SET NX PX`），简单高效         |
| 对安全性要求较高，但能接受一定复杂度和性能开销 | **Redlock**（例如使用 Redisson 或 go-redsync）               |
| 金融级绝对互斥，不能有任何并发冲突             | 放弃 Redis，使用 **etcd** 或 **ZooKeeper**（提供线性一致性和 fencing token） |

### 八、代码示例（简单实现）

#### Go（go-redsync 实现）

```go
import (
    "github.com/go-redsync/redsync/v4"
    "github.com/go-redsync/redsync/v4/redis/goredis/v9"
    goredislib "github.com/redis/go-redis/v9"
)

func main() {
    client1 := goredislib.NewClient(&goredislib.Options{Addr: "node1:6379"})
    client2 := goredislib.NewClient(&goredislib.Options{Addr: "node2:6379"})
    client3 := goredislib.NewClient(&goredislib.Options{Addr: "node3:6379"})
    pool1 := goredis.NewPool(client1)
    pool2 := goredis.NewPool(client2)
    pool3 := goredis.NewPool(client3)

    rs := redsync.New(pool1, pool2, pool3) // 必须传入所有节点池

    mutex := rs.NewMutex("my-lock")
    if err := mutex.Lock(); err != nil {
        panic(err)
    }
    defer mutex.Unlock()
    // 业务逻辑
}
```

### 九、红锁 vs etcd/ZooKeeper

| 特性             | Redlock (Redis)            | etcd / ZooKeeper                                 |
| :--------------- | :------------------------- | :----------------------------------------------- |
| 一致性模型       | 不保证线性一致性，依赖时钟 | 线性一致性（Raft / ZAB）                         |
| 防 GC / 网络延迟 | 无 fencing token           | 支持 fencing token（单调递增的 revision / zxid） |
| 性能             | 较高（内存操作）           | 较低（需要磁盘落盘和共识）                       |
| 部署复杂度       | 需要多个独立 Redis 节点    | 需要 3/5 节点集群                                |
| 安全性           | 有争议，极端情况可能失效   | 业界公认强安全                                   |

------

### 十、总结

**红锁是 Redis 官方提出的、通过多独立节点多数派投票来提升锁安全性的算法。它解决了主从异步复制丢锁问题，但仍有时钟跳变、GC 暂停、节点重启等隐患，并非绝对安全的分布式锁。在 Go 语言中，`go-redsync` 是其标准实现；若业务对互斥有极致要求，应优先选用 etcd 或 ZooKeeper。**
